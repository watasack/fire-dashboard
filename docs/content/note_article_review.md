### [HIGH] 自己紹介と開発ストーリーの内容が重複している

**対象:** `note_self_intro.md`、`note_tool_story.md`

両記事が同じ骨格で書かれており、読者が2記事目を読む動機が生まれにくい。

| 重複している要素 | 自己紹介 | 開発ストーリー |
|---|---|---|
| 「3年かけて作った」 | あり | あり |
| 「なかったから作った」 | あり | あり |
| 「妻の育休がきっかけ」 | あり | あり |
| ツール機能の概要説明 | あり | あり |

**販売への影響:** 自己紹介→開発ストーリーと読み進んだ読者が「同じ話だ」と感じると、メイン分析記事への到達率が下がる。販売記事まで読まれなくなるリスクがある。

**対応案（どちらかを選択）:**
- **A. 統合する:** 2記事を1記事にまとめ「自己紹介兼開発ストーリー」とする。重複する内容を削除してコンパクトに。
- **B. 切り口を明確に分ける:** 自己紹介は「著者の属性と信頼性」に絞り、開発ストーリーは「ツールで実際に変わった意思決定の具体例」を中心にする。現状の開発ストーリーにある支出見直し・NISA積立額の決定などの実例部分を強化する。

### [MEDIUM] 投稿順序の再検討

**現状の順序:** 自己紹介 → 開発ストーリー → 分析記事 → 販売記事

noteとSNSの流入特性を考えると、この順序が最適とは限らない。

**問題点:**
- 自己紹介・開発ストーリーは「すでに著者を知っている人」向けのコンテンツ。初見の読者には刺さりにくい。
- 分析記事（「育休でFIREが何年遅れるか」）が最も拡散されやすいタイトルであり、検索・SNS流入の入口になりやすい。

**検討すべき代替順序:**
```
分析記事（バズ入口） → 自己紹介/開発ストーリー（フォロワー化） → 販売記事
```

X投稿で分析記事を最初に露出させ、そこから著者に興味を持った人が自己紹介・開発ストーリーを読む流れのほうが、最終的な販売記事への到達数が増える可能性がある。

**対応:** `docs/plans/monetization_strategy.md` の投稿スケジュールと合わせて検討する。

---

### [MEDIUM] 無料デモ→有料版の転換導線が弱い

**対象:** `note_sales_article.md`

比較表（無料ツール vs このシミュレーター）はあるが、「無料デモを試した後、なぜフル版が必要になるか」という体験の流れが書かれていない。

**現状の問題:**
- 無料デモで「FIRE到達年齢」が分かる
- しかし育休・産休・モンテカルロはフル版のみ
- この「無料デモで気づく限界」が読者に伝わっていない

**対応案:** 販売記事に以下のような一段を追加する。

> 無料デモを試すと「FIRE到達年齢：46歳」のような数字が出ます。ただし、このデモは「育休なし・フルタイム継続」の前提で動いています。「育休を取ったら？」「時短勤務になったら？」「子どもが2人になったら？」——こういう現実的な問いに答えるのがフル版です。

### [LOW] 分析記事の画像コメントが記事本文に残る

**対象:** `note_article_draft.md`

`<!-- 📷 画像: ... -->` 形式のHTMLコメントはnoteエディタ上では見えないが、管理上の混乱を避けるために投稿前に整理する。使用する画像が確定したら `docs/images/note/` に確定コピーし、コメントを削除または画像に置き換える。
