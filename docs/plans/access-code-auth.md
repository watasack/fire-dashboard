# アクセスコード認証 技術設計

作成日: 2026-04-04

関連: [monetization_strategy.md](./monetization_strategy.md)

---

## 概要

note 有料記事の購入者のみがフル版を利用できるよう、アクセスコードによる機能分岐を実装する。
デモ版とフル版は**同一コードベース・同一URL**で提供し、コード入力によって解放する。

---

## 運用フロー

```
① 事前にコード100個を生成（tools/generate-codes.ts を1回実行）
② 生成したコード一覧を手元のスプレッドシートで管理
③ Vercel 環境変数 VALID_CODES に100個全て登録
④ note で購入 → 購入者から DM / メールで連絡
⑤ スプレッドシートから未送付のコードを1つ選んで返信
⑥ 購入者がサイトでコード入力 → API 検証 → フル版解放
```

### 管理スプレッドシートのフォーマット

| コード | 送付日 | 購入者メモ |
|--------|--------|-----------|
| fire-a1b2c3 | — | 未送付 |
| fire-d4e5f6 | 2026-04-01 | note ユーザー A さん |

コードの補充が必要になったら `tools/generate-codes.ts` を再実行し、
Vercel 環境変数を更新して再デプロイする（数分）。

## アーキテクチャ

```
ブラウザ                    Vercel (Edge)
─────────────────────────────────────────
[コード入力UI]
  │ POST /api/validate {code}
  │──────────────────────────────────────→
  │                         env: VALID_CODES と照合
  │                         一致 → token = `${code}.${HMAC(SECRET, code)}`
  │←──────────────────────────────────────
  │ {token}
  │
  │ localStorage.setItem("fire_token", token)
  │
[ページロード]
  │ GET /api/ping  Authorization: Bearer {token}
  │──────────────────────────────────────→
  │                         token を "{code}.{hmac}" に分割
  │                         HMAC(SECRET, code) を再計算
  │                         timingSafeEqual で署名を検証
  │                         code が VALID_CODES に含まれるか確認
  │←──────────────────────────────────────
  │ {valid: true/false}
  │
  │ valid === true → フル版表示
  │ valid === false → デモ版表示 + コード入力を促す
```

### トークン形式の詳細

コードをサーバーで保存せずに検証できるよう、トークンにコード自体を埋め込む。

```
token = "{code}.{HMAC-SHA256(AUTH_SECRET, code)}"
例:   "fire-a1b2c3.8f3d9e2a1b..."
```

`/api/ping` の検証ロジック:
1. トークンを `.` で分割 → 要素数が 2 でなければ即 `{ valid: false }` を返す
2. `HMAC(AUTH_SECRET, code)` を再計算し `expected` を得る
3. `sig.length !== expected.length` なら即 `{ valid: false }` を返す（長さガード）
4. `timingSafeEqual(Buffer.from(sig, "hex"), Buffer.from(expected, "hex"))` でタイミング攻撃を防ぎながら署名を比較
5. `VALID_CODES` にコードが含まれるか確認（無効化済みコードの検知）

> **注意**: ステップ1・3のバリデーションを省略すると、不正なトークンで `timingSafeEqual` が `TypeError` を投げて `/api/ping` が 500 エラーになる。必ず長さガードを入れること。

### ローディング状態

`/api/ping` のレスポンス待ち中は**フル版でも デモ版でもない「認証確認中」状態**を表示する。
フル版ユーザーに一瞬デモ版が見えるちらつきを防ぐ。

```typescript
type AuthState = "loading" | "authed" | "demo"
// loading: スケルトン or スピナー表示（全タブをグレーアウト）
// authed:  フル版表示
// demo:    デモ版表示（ロックオーバーレイ）
```

`/api/ping` は Edge Function のため通常 100ms 以内に応答する。

### ネットワーク障害時の挙動

**フェールオープン（障害時にフル版表示）を採用する。**

理由: フェールクローズにすると、正規購入者がネットワーク障害のたびにデモ版に
切り替わり、「お金を払ったのに使えない」という体験が生まれる。
不正利用のリスクより購入者の体験を優先する（副業規模での割り切り）。

```typescript
// /api/ping が失敗（タイムアウト・ネットワークエラー）した場合
// localStorage にトークンが存在すれば → フル版表示
// localStorage にトークンが存在しなければ → デモ版表示
```

## UI 設計方針

### デモ/フル版の切り替え方式

**採用: `isDemoMode` props + ロックオーバーレイ**

```typescript
// AccessGate の render ロジック
if (authState === "loading") return <DashboardSkeleton />;  // ← loading 中は Dashboard をレンダーしない
<FireDashboard isDemoMode={authState === "demo"} />         // ← boolean に変換して渡す
```

```
app/page.tsx
  └── <AccessGate>                    ← authState: "loading" | "authed" | "demo" を管理
        ├── authState === "loading" → <DashboardSkeleton />（全タブグレーアウト）
        └── authState !== "loading" → <FireDashboard isDemoMode={authState === "demo"} />
              ├── <ConfigPanel isDemoMode />
              │     ├── 基本タブ        → 解放（年収・支出・資産）
              │     ├── 収入タブ        → 🔒 ロックオーバーレイ
              │     ├── 投資タブ        → 🔒 ロックオーバーレイ
              │     ├── ライフタブ      → 🔒 ロックオーバーレイ
              │     └── 詳細タブ        → 🔒 ロックオーバーレイ
              ├── 資産推移グラフ        → ぼかし表示
              ├── 収支・年次表タブ      → 🔒 ロックオーバーレイ
              └── シナリオ比較          → 🔒 ロックオーバーレイ
```

**Context を使わない理由**: props のバケツリレーは `FireDashboard` → `ConfigPanel` の
1段のみ。Context の追加はこの規模ではオーバーエンジニアリング。

**別ページ（`/app`）にしない理由**: デモ版とフル版で UI は完全に同一にしたい。
機能追加のたびに2箇所直す保守コストを避ける。

### コード入力UI の配置

**採用: ロックオーバーレイ内にコード入力欄を配置**

```
┌──────────────────────────────────┐
│  収入タブのコンテンツ（ぼかし）   │
│  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  │
│  ░░░░░░░░░░░░░░░░░░░░░░░░░░░░░  │
│  ┌────────────────────────────┐  │
│  │ 🔒 フル版で利用できます     │  │
│  │ アクセスコードを入力        │  │
│  │ [____________] [解放する]   │  │
│  └────────────────────────────┘  │
└──────────────────────────────────┘
```

モーダルにしない理由: デモを体験させる前に壁を作ると離脱率が上がる。
バナーにしない理由: 見落とされやすく「何ができないか」が伝わらない。
ロックオーバーレイの利点: 「触ったときに初めてロックに気づく」体験が
「育休の設定がしたいのにできない」という購買動機を生む。

## 環境変数

| 変数名 | 説明 | 例 |
|--------|------|----|
| `VALID_CODES` | 有効なコードのカンマ区切りリスト（サーバーサイドのみ） | `fire-a1b2,fire-c3d4,...` |
| `AUTH_SECRET` | HMAC 署名用シークレット（32文字以上推奨） | `random-secret-string` |

`NEXT_PUBLIC_` プレフィックスを付けないことで、クライアントバンドルに露出しない。

## デモ版の制限仕様

フル版との機能差分（`access-gate.tsx` で制御）:

| 機能 | デモ版 | フル版 |
|------|--------|--------|
| 基本入力（年収・支出・資産） | ○ | ○ |
| 配偶者・育休・時短の詳細設定 | × | ○ |
| 投資・NISA/iDeCo 設定 | × | ○ |
| ライフ・教育費設定 | × | ○ |
| 資産推移グラフ | ぼかし表示 | ○ |
| 収支グラフ・年次表 | × | ○ |
| モンテカルロ成功確率 | ○（参考値・注1） | ○ |
| シナリオ比較 | × | ○ |
| URL シェア | × | ○ |

> **注1**: デモ版のモンテカルロ成功確率は基本入力3項目（年収・支出・資産）のみを使った
> 簡易計算値であり、フル版（育休・教育費・住宅ローン等を反映）とは異なる。
> UIには「詳細設定を反映するにはフル版が必要です」と明示する。

## セキュリティ上の割り切り

- **コードの共有は防止しない**: 1コードを複数人で使い回されても追わない。副業規模では許容範囲。
- **コード無効化**: 環境変数から削除して再デプロイ。即時反映（次のページロード時に `/api/ping` で検知）。
- **`/api/validate` の総当たり攻撃対策**: Vercel の Edge Function はデフォルトでレートリミットが効く。必要なら `rate-limit` ミドルウェアを追加。
- **`/api/ping` のレートリミット**: ページロードごとに呼ばれるため、Vercel の無料枠（月100万リクエスト）に注意。悪意ある連打は Vercel のデフォルト保護に委ねる（副業規模では十分）。

## 不正使用の監視（オプション）

副業規模では完全な追跡は費用対効果が低い。問題が起きたときだけ対応する方針を推奨。

### レベル1：Vercel ログで目視確認（追加実装ゼロ）

Vercel ダッシュボードの Functions ログから `/api/validate` のアクセスを確認できる。
同じコードが短期間に多数の IP から叩かれていれば共有を検知できる。

**限界**: 無料プランではログ保持が 24〜72 時間程度。

### レベル2：APIルートでのログ出力（推奨・軽微な実装）

`/api/validate` に数行追加するだけ。Vercel のログに残り、問題発生時に遡って確認できる。

```typescript
// app/api/validate/route.ts に追加
console.log(JSON.stringify({
  event: "validate",
  code: code.slice(0, 6) + "***",  // コード全文は出さない
  ip: request.headers.get("x-forwarded-for"),
  ua: request.headers.get("user-agent"),
  ts: new Date().toISOString(),
}));
```

普段は確認不要。SNS 拡散などの問題が発生したときだけ Vercel ダッシュボードを開いて目視する。

### レベル3：Axiom 連携（本格ログ・無料枠あり）

Vercel と [Axiom](https://axiom.co/) を連携するとログが長期保存され、クエリ検索が可能になる。
「このコードが過去30日で何件・何 IP から使われたか」を即座に確認できる。

- 設定工数: 数時間
- 費用: 無料枠（月100GB）で十分
- 導入タイミング: 月20件以上の販売が継続したら検討

### 検知後の対応

| 状況 | 対応 |
|------|------|
| 数人での使い回し | 許容する（副業規模では追わない） |
| SNS でコードが拡散された | 環境変数から削除 → 再デプロイで即無効化 |
| 正規購入者への影響 | 新しいコードを発行して送り直す |

## 今後の拡張（件数が増えた場合）

現状の Vercel 環境変数方式では、コード追加のたびに再デプロイが必要。
月50件以上になったら Notion データベース + Notion API への移行を検討する。

```
現状: VALID_CODES 環境変数 → 再デプロイで更新
移行後: Notion DB → API ルートが都度参照 → 再デプロイ不要
```
