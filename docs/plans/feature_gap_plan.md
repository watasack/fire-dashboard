## 設計方針（ユーザー視点）

### 1. 収入入力：年収（税引き前）に移行【最優先・大改修】

**理由**: 日本人が「自分の収入」として認識しているのは年収。手取り月収は毎月変動し、
自分で計算するには税率・社保の知識が必要で、ユーザーに不要な認知負荷を与えている。
育休給付金も本来「標準報酬月額（税引き前）の67%」が正確な計算式であり、
年収入力への移行で計算精度も向上する。

**UI設計**:
```
あなたの年収（税引き前）  [500] 万円
  → 手取り目安: 約390万円（月約33万円）※自動計算して参考表示
```

**計算内容**:
- 累進課税（所得税）: 基礎控除・配偶者控除・扶養控除を考慮
- 社会保険料: 健康保険（9.98%）・厚生年金（18.3%）を事業主折半で計算
- 雇用形態別に計算ロジックを分岐（会社員/個人事業主/専業主婦）
- 育休給付金 = 標準報酬月額 × 67%（前半180日）、50%（後半）に修正

**影響範囲**: `full_app.py`、`src/simulator.py` 全体（最大の改修）

---

### 2. FIRE後の収入入力（サイドFIRE）【高優先】

**理由**: 「完全FIRE」より「セミFIRE（しばらく副業して段階的にリタイア）」の
ニーズが大きい。全競合（1・3・4）が対応済み。現状はFAQで「生活費から副業収入を
引いて入力してください」と案内しているが、UIとして明示するべき。

**UI設計**:
```
FIRE後の収入（任意）
  [60] 歳まで　月 [5] 万円
```
終了年齢 + 月額の2項目のみ。複数期間の設定は過剰。

**計算内容**:
- FIRE年齢〜終了年齢の間、月収入として支出から差し引く
- 終了年齢以降は収入ゼロとして計算

**影響範囲**: `full_app.py`、`src/simulator.py`

---

### 3. 教育方針の選択【高優先】

**理由**: 現状の「公立固定」は実態と合わない。競合3（ふくわら式）は学歴・学制まで
細かく設定できる。最低でも「大学のみ私立」という日本で最も多いパターンを追加すべき。

**UI設計**:
```
教育方針（全員共通）
  ○ 公立メイン      （公立小中高・国立大）
  ● 大学のみ私立    （公立小中高・私立大）← デフォルト推奨
  ○ 私立メイン      （私立一貫・私立大）
```

**教育費テーブル（万円/年）**:

| 段階 | 公立メイン | 大学のみ私立 | 私立メイン |
|------|-----------|------------|-----------|
| 幼稚園 | 30 | 30 | 50 |
| 小学校 | 35 | 35 | 150 |
| 中学校 | 50 | 50 | 150 |
| 高校 | 50 | 50 | 100 |
| 大学 | 120 | 150 | 150 |

**影響範囲**: `full_app.py`、`config.yaml`、`src/simulator.py`

---

### 4. FIRE後の税金計算【高優先】

**理由**: FIRE直後は前年の給与所得をベースに国民健康保険料が計算されるため、
初年度に大きなコストスパイクが発生する。これを無視するとFIRE後の収支が楽観的になる。
競合2（63400r）が実装済みで、ユーザーが「FIRE後にこんなに税金がかかるとは知らなかった」
という誤解を防ぐためにも重要。

**計算内容**:
- FIRE後の金融所得に対する20.315%課税（含み益割合に応じて計算）
- 国民健康保険料（前年所得ベース）
- 住民税（前年所得ベース）
- FIRE初年度のスパイクを個別計算（前年給与が高い場合に国保が大きく跳ね上がる）

**注意**: ①（年収入力への移行）と連動する。①完了後に実装するとロジックが一貫する。

**影響範囲**: `src/simulator.py`（FIRE後費用計算）、`full_app.py`

---

### 5. 成功確率 ＋ 最悪ケース枯渇年齢【中優先】

**理由**: 「成功確率80%」は抽象的でユーザーが「20%の失敗」をイメージしにくい。
競合1・4・5が資産枯渇年齢を表示しており、具体的な数字として見せることが有効。

**UI設計**:
```
✅ 1,000通りのうち 800通り で、90歳まで資産が持ちました
⚠️  最悪ケースでも [XX] 歳まで資産が持ちます（下位10%シナリオ）
```
- 成功確率（現在の指標）をメインに維持
- 悲観シナリオ（10パーセンタイル）の枯渇年齢を補足表示
- 枯渇しない場合は「シミュレーション終了（90歳）時点でも資産あり」と表示

**影響範囲**: `full_app.py`、`src/simulator.py`

---

### 6. 年次収支テーブル【中優先】

**理由**: 「何歳の時にいくら使うか」「ローン完済後に支出がどう減るか」
「年金が入ってから収支がどう変わるか」を年齢ごとに確認したいニーズがある。
競合1・3が対応済み。モンテカルログラフとは役割が異なり補完関係。

**UI設計**:
- 中央値（50パーセンタイル）の1本のみ表示（複数パーセンタイルは混乱を招く）
- 表示列: 年齢 / 資産残高 / 年収（手取り） / 支出 / 純CF
- `st.dataframe` でスクロール可能なテーブルとして表示

**モンテカルログラフとの役割分担**:
| 表示 | 役割 |
|------|------|
| モンテカルログラフ（1,000通り） | リスクの全体像・成功確率 |
| 年次収支テーブル（中央値1本） | 家計の年齢別タイムライン・具体的な数字確認 |

**影響範囲**: `full_app.py`、`src/simulator.py`（年次データの返却追加）

---

### 7. 収支グラフ（収入・支出・純CF）【中優先】

**理由**: 「いつ収入が支出を下回るか」「FIRE後の収支バランス」が
棒グラフで視覚的にわかる。競合1が対応済み。年次テーブルと同じデータの別ビュー。

**UI設計**:
- 中央値シナリオの grouped bar chart（収入・支出・純CF）
- 5年間隔で表示
- 年次テーブルと同じタブに配置

**影響範囲**: `full_app.py`

---

### 8. 取り崩し戦略の選択【中優先】

**理由**: 競合2（63400r）は3種類の取り崩し戦略を提供。
現状は固定支出のみで、FIRE後に「資産が増えたら支出を増やす」「減ったら抑える」
という現実的な行動が反映できない。

**UI設計**:
```
取り崩し戦略
  ● 固定額（毎月 XX 万円）      ← 現状・デフォルト
  ○ 資産の一定割合（年 X.X%）
  ○ ガードレール（資産に応じて自動調整）
```

**影響範囲**: `src/simulator.py`（FIRE後支出ロジック）、`full_app.py`

---

### 9. NISA/iDeCo対応【中優先】

**理由**: 競合3（ふくわら式）は投資口座の種類（特定口座・NISA・iDeCo）と
取り崩し時の売却順位まで設定できる。現状のこちらのツールはNISA残高を0として扱っており、
NISAを活用しているユーザーにとっては資産の過小評価になっている。

**現状の問題**:
- NISAの既存残高 = 0として計算（FAQに明記済みだが不満が出る可能性あり）
- iDeCoは60歳まで引き出せない制約を考慮していない

**実装方針（案）**:
```
資産の内訳（任意）
  NISA残高    [300] 万円  ← 非課税で取り崩せる分
  iDeCo残高   [100] 万円  ← 60歳以降に解放
  一般口座     [自動計算] 万円
```
- NISA残高は取り崩し時に課税なし（20%課税の対象外）として計算
- iDeCo残高は60歳以降のみ取り崩し可能として計算

**注意**: ①（年収入力）・④（FIRE後税金）と連動する。実装はPhase 2後半が適切。

**影響範囲**: `full_app.py`（入力追加）、`src/simulator.py`（取り崩し計算の分岐）

---

### 10. MCの精度向上【低〜中優先】

**現状**: 正規分布 + 平均回帰モデル

**競合の手法**:
- 競合2: Student-t分布（自由度12）+ コレスキー分解による相関行列
- 競合4（カネグリ）: **ブートストラップ法**（1971〜2022年の実績52年分から無作為抽出）

**カネグリのブートストラップ法が優れている点**:
- 正規分布の仮定が不要
- 1971〜2022年に実際に起きた暴落（オイルショック・リーマン・コロナ）をそのまま再現
- 株・債券・為替・インフレを同じ年のデータで使用するため資産間相関が歴史的に正確
- ユーザーに説明しやすい（「過去52年間のデータを使いました」）

**判断基準**: 現在の正規分布でも十分な精度はあるため、他の優先項目完了後に検討。

---

### 10. マルチアセット対応【低優先】

**理由**: 競合2・4は株式以外の資産クラス（債券・金ETF等）に対応。
ただし、こちらのターゲット（育休世代の共働き世帯）は
インデックス投資（全世界株・S&P500）が主流であり、
複雑なアセットアロケーションを必要とするユーザーは少ない。

**判断基準**: 購入者から要望が来たら対応。現状は不要。

---

### 11. PDF出力【低優先】

**判断基準**: 購入者から要望が来たら対応。現状は不要。

---

### 12. 改善提案・最適化【低優先】

**判断基準**: Phase 1〜2が完了し、ユーザー需要が確認できてから対応。

## 実装順序

```
Phase 1（ユーザー体験の根本改善）:
  ① 年収（税引き前）入力への移行      ← 最大の改修・最大の効果・他に影響
  ② FIRE後の収入入力（サイドFIRE）    ← 少ない変更で高い需要対応
  ③ 教育方針の3択                    ← 設定の現実感が上がる

Phase 2（計算精度と可視化の強化）:
  ④ FIRE後の税金計算（国保スパイク等） ← ①と連動・実態に即した計算
  ⑤ 最悪ケース枯渇年齢の表示          ← 成功確率の補足
  ⑥ 年次収支テーブル                  ← 具体的な年齢別タイムライン
  ⑦ 収支グラフ                        ← ⑥と同データの別ビュー
  ⑧ 取り崩し戦略の選択               ← FIRE後の現実的な行動を反映
  ⑨ NISA/iDeCo対応                   ← ④と連動・取り崩し課税の正確化

Phase 3（要検討）:
  ⑩ MCの精度向上（ブートストラップ法） ← カネグリ方式・説明しやすい
  ⑪ マルチアセット対応               ← ターゲット層には優先度低
  ⑫ PDF出力
  ⑬ 改善提案・最適化
```

## 注意事項

**全体方針: デモ版（`demo_app.py`）への反映について**
`demo_app.py` の各種機能アップデートについては、`full_app.py` 向けのすべての機能追加・改修が完了した後に一括して着手します。各機能の実装フェーズ（Phase 1〜）においては、常に `full_app.py` と `src/` 配下のロジックのみを対象として開発を進めます。

**①の「年収入力への移行」は他のすべてに影響する。**
育休給付金・社保・税金の計算ロジックが連動して変わるため、
②〜③より先に設計を確定させること。
①を後回しにして②〜③を実装すると、①実装時に再修正が発生する。

**④のFIRE後税金計算は①と連動する。**
年収入力が完了してから実装するとロジックが一貫する。

**⑨のブートストラップ法は「過去の実績データをそのまま使う」ため、
将来の市場が過去と大きく異なる場合（例：長期デフレ・超高インフレ）には
過去データに引きずられるリスクがある。**
正規分布との使い分けをユーザーが選べる設計も一案。
