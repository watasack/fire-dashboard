## 設計方針（ユーザー視点）

### 1. 収入入力：年収（税引き前）に移行【最優先・大改修】

**理由**: 日本人が「自分の収入」として認識しているのは年収。手取り月収は毎月変動し、
自分で計算するには税率・社保の知識が必要で、ユーザーに不要な認知負荷を与えている。
育休給付金も本来「標準報酬月額（税引き前）の67%」が正確な計算式であり、
年収入力への移行で計算精度も向上する。

**UI設計**:
```
あなたの年収（税引き前）  [500] 万円
  → 手取り目安: 約390万円（月約33万円）※自動計算して参考表示
```

**計算内容**:
- 累進課税（所得税）: 基礎控除・配偶者控除・扶養控除を考慮
- 社会保険料: 健康保険（9.98%）・厚生年金（18.3%）を事業主折半で計算
- 雇用形態別に計算ロジックを分岐（会社員/個人事業主/専業主婦）
- 育休給付金 = 標準報酬月額 × 67%（前半180日）、50%（後半）に修正

**影響範囲**: `full_app.py`、`src/simulator.py` 全体（最大の改修）

---

### 2. FIRE後の収入入力（サイドFIRE）【高優先】

**理由**: 「完全FIRE」より「セミFIRE（しばらく副業して段階的にリタイア）」の
ニーズが大きい。現状はFAQで「生活費から副業収入を引いて入力してください」と
案内しているが、UIとして明示するべき。

**UI設計**:
```
FIRE後の収入（任意）
  [60] 歳まで　月 [5] 万円
```
終了年齢 + 月額の2項目のみ。複数期間の設定は過剰。

**計算内容**:
- FIRE年齢〜終了年齢の間、月収入として支出から差し引く
- 終了年齢以降は収入ゼロとして計算

**影響範囲**: `full_app.py`、`src/simulator.py`

---

### 3. 教育方針（3択）【高優先】

**理由**: 現状の「公立固定」は実態と合わない場合がある。
「公立か私立か」の二択も不十分で、日本で最も多いパターンは
「小中高は公立、大学だけ私立」。

**UI設計**:
```
教育方針（全員共通）
  ○ 公立メイン      （公立小中高・国立大）
  ● 大学のみ私立    （公立小中高・私立大）← デフォルト推奨
  ○ 私立メイン      （私立一貫・私立大）
```

**教育費テーブル（万円/年）**:

| 段階 | 公立メイン | 大学のみ私立 | 私立メイン |
|------|-----------|------------|-----------|
| 幼稚園 | 30 | 30 | 50 |
| 小学校 | 35 | 35 | 150 |
| 中学校 | 50 | 50 | 150 |
| 高校 | 50 | 50 | 100 |
| 大学 | 120 | 150 | 150 |

**影響範囲**: `full_app.py`、`config.yaml`、`src/simulator.py`

---

### 4. 成功確率 ＋ 最悪ケース枯渇年齢【中優先】

**理由**: 「成功確率80%」は抽象的でユーザーが「20%の失敗」をイメージしにくい。
枯渇年齢を補足することで「最悪でもどこまで持つか」が具体的にわかる。

**UI設計**:
```
✅ 1,000通りのうち 800通り で、90歳まで資産が持ちました
⚠️  最悪ケースでも [XX] 歳まで資産が持ちます
```
- 成功確率（現在の指標）をメインに維持
- 悲観シナリオ（10パーセンタイル）の枯渇年齢を補足表示
- 枯渇しない場合は「シミュレーション終了（90歳）時点でも資産あり」と表示

**影響範囲**: `full_app.py`、`src/simulator.py`

---

### 5. 年次収支テーブル【中優先】

**理由**: 「何歳の時にいくら使うか」「ローン完済後に支出がどう減るか」
「年金が入ってから収支がどう変わるか」を年齢ごとに確認したいニーズがある。
モンテカルログラフとは役割が異なり、補完関係にある。

**UI設計**:
- 中央値（50パーセンタイル）の1本のみ表示（複数パーセンタイルは混乱を招く）
- 表示列: 年齢 / 資産残高 / 年収（手取り） / 支出 / 純CF
- `st.dataframe` でスクロール可能なテーブルとして表示

**モンテカルログラフとの役割分担**:
| 表示 | 役割 |
|------|------|
| モンテカルログラフ（1,000通り） | リスクの全体像・成功確率 |
| 年次収支テーブル（中央値1本） | 家計の年齢別タイムライン・具体的な数字確認 |

**影響範囲**: `full_app.py`、`src/simulator.py`（年次データの返却追加）

---

### 6. 収支グラフ（収入・支出・純CF）【中優先】

**理由**: 「いつ収入が支出を下回るか」「FIRE後の収支バランス」が
棒グラフで視覚的にわかる。年次テーブルと同じデータの別ビュー。

**UI設計**:
- 中央値シナリオの grouped bar chart（収入・支出・純CF）
- 5年間隔で表示（競合と同様）
- 年次テーブルと同じタブに配置

**影響範囲**: `full_app.py`

---

### 7. PDF出力【低優先】

**判断基準**: 購入者から要望が来たら対応。現状は不要。

---

### 8. 改善提案・最適化【低優先】

**判断基準**: Phase 1〜2が完了し、ユーザー需要が確認できてから対応。

---

### 7. 取り崩し戦略の選択【中優先】（競合2より）

**理由**: 競合2は固定額・%指定・ガードレール＋U字型の3種類を提供。
現状は固定支出のみで、FIRE後に「資産が増えたら支出を増やす」「減ったら抑える」
という現実的な行動が反映できない。

**UI設計（案）**:
```
取り崩し戦略
  ● 固定額（毎月 XX 万円）      ← 現状
  ○ 資産の一定割合（年 X.X%）
  ○ ガードレール（資産に応じて自動調整）
```

**影響範囲**: `src/simulator.py`（FIRE後支出ロジック）、`full_app.py`

---

### 8. FIRE後の税金計算【中優先】（競合2より）

**理由**: FIRE直後は前年の給与所得をベースに国民健康保険料が計算されるため、
初年度に大きなコストスパイクが発生する。これを無視するとFIRE後の収支が楽観的になる。

**計算内容（案）**:
- FIRE後の金融所得に対する20.315%課税
- 国民健康保険料（前年所得ベース）
- 住民税（前年所得ベース）
- FIRE初年度のスパイクを個別計算

**影響範囲**: `src/simulator.py`（FIRE後費用計算）

## 実装順序

```
Phase 1（ユーザー体験の根本改善）:
  ① 年収（税引き前）入力への移行      ← 最大の改修・最大の効果
  ② FIRE後の収入入力（サイドFIRE）    ← 少ない変更で高い需要対応
  ③ 教育方針の3択                    ← 設定の現実感が上がる

Phase 2（可視化・計算精度の強化）:
  ④ 最悪ケース枯渇年齢の表示          ← 成功確率の補足として追加
  ⑤ 年次収支テーブル
  ⑥ 収支グラフ
  ⑦ 取り崩し戦略の選択               ← 競合2より
  ⑧ FIRE後の税金計算                 ← 競合2より（①と連動）

Phase 3（要検討）:
  ⑨ Student-t分布への変更            ← 競合2より・モデル精度向上
  ⑩ マルチアセット対応               ← 競合2より・対象ユーザーが限定的
  ⑪ PDF出力
  ⑫ 改善提案・最適化
```

---

## 注意事項

**①の「年収入力への移行」は他のすべてに影響する**。
育休給付金・社保・税金の計算ロジックが連動して変わるため、
②〜③より先に設計を確定させること。
①を後回しにして②〜③を実装すると、①実装時に再修正が発生する。

**⑧のFIRE後税金計算は①と連動する**。
年収入力が完了してから実装するとロジックが一貫する。
