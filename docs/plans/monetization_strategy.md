# FIREシミュレーター サービス化戦略（副収入3万円特化版）

作成日: 2026-03-04  最終更新: 2026-03-05

---

## 概要

**目標：月3万円の副収入を、本業を圧迫せずに安定して得る**

- ビジネスモデル：note有料記事の購入 → Webツール（Streamlit）のアクセスコード発行
- 販売プラットフォーム：note 有料記事
- 集客：X/note（短期）＋ SEO（中長期）・キーワード「共働き FIRE」「育休 FIRE」
- サポート：メールのみ・週1回まとめて対応
- アクセスコード：購入者ごとに個別発行

**note一本化の理由（Gumroadを使用しない理由）：**
- ユーザーに「記事・読み物を買う」という文脈で信頼を構築した上でツールを提供できる
- note内での回遊性が高く、SEO記事から有料記事への誘導がスムーズ
- 日本のFIRE層との親和性が高く、決済の心理的ハードルが低い
- 販売プラットフォームを一本化することで、ドキュメント管理やサポートの工数を最小化する

**単一価格を維持する理由（2価格モデルをやらない理由）：**
- 迷わせると買わない。選択肢が増えるとCV率が下がるリスクがある
- 無料Streamlitデモがすでに「Lite相当」の役割を果たしている
- 2価格モデルは保守コストが倍になる（副業モデルでは致命的）
- Phase 2（3ヶ月後・販売データが取れてから）で再検討する

**このモデルで捨てるもの：**
- SaaS化・サブスクリプション（現フェーズでは追わない）
- 機能の高度化（3ヶ月間は追加禁止）
- Docker版・Tier 3（エンジニア向けは後回し）
- SNSシェア機能（Phase 2で検討）
- 2価格モデル（Phase 2でデータを見てから判断）

## ツールの強み（競合優位性）

- **モンテカルロシミュレーション**：1,000回試行による確率的FIRE到達時期の可視化
- **育休・産休・時短の完全反映**：日本固有の事情をそのまま入力できる唯一のツール
- **最悪シナリオの可視化**：「リターンが悪い年が続いたら？」を定量的に示す
- **夫婦別収入設定**：会社員 × 個人事業主など非対称な世帯設定

> 感度分析・年金最適化・GARCHは技術的差別化として存在するが、
> 副収入フェーズでは訴求に使わない（サポート工数が増えるため）。
> 将来のSaaS化フェーズで改めて展開する。

## 信頼設計（最重要・現戦略の最大の欠落）

金融系ツールは「誰が作ったか」が購買決定に直結する。
プロダクトの品質よりも**発信者への信頼**が支配的。

### 発信者プロフィールとして出す情報

- 宇宙・航空系エンジニア（技術的信頼性）
- 機械学習・数理モデリングの実務経験
- 自分自身がFIREを本気で計画している当事者
- 家族構成：共働き・子供あり（ペルソナと同じ境遇）

### 出さないこと

- 金融の専門家・アドバイザーを名乗らない
- 「投資をすすめる」表現を使わない
- 「数理モデリングの専門家が作った、自分用シミュレーター」として立つ

> 法的注意：金融商品取引業登録なしに「投資アドバイス」を有償提供することは
> 金融商品取引法違反。「シミュレーターツールの販売」として明確に位置づける。

### 体験的証拠として使える要素（プロフィール・発信で具体化する）

プロフィールだけでは信頼が足りない。「この人は本当にやっている」と感じさせる体験的事実を出す。

- 自分の資産額・貯蓄率（出せる範囲で。伏せ字可）
- 何年間シミュレーションを回し続けているか
- シミュレーション結果が自分の意思決定をどう変えたか（育休取得・支出削減等）
- 子供が生まれてFIRE計画がどう変わったかの実体験

「このツールを一番長く使っている当事者が自分」であることが最大の信頼根拠。

## 商品設計（副収入特化・シンプル版）

### 商品は1つだけ：Webツールのアクセスコード

**価格を複数にしない理由：迷わせると買わない。**

| 含まれる機能 | 含まれない機能（将来フェーズ向け） |
|------------|----------------------------------|
| MCシミュレーション（1,000回） | 感度分析 |
| FIRE到達確率の表示 | 年金受給最適化（繰り上げ/繰り下げ比較） |
| 最悪ケース資産推移グラフ | GARCHモデル |
| 育休・産休・時短勤務の反映 | Docker版 |
| 夫婦別収入入力 | |

**機能を絞る理由：**
- 機能が増えるほどサポート問い合わせが増える（本業を圧迫する）
- 年金最適化を入れると「繰下げの前提は？加給年金は？」等の専門質問が来る
- 「育休を正確に反映できる」だけで他の無料ツールと十分差別化できる

### 3ヶ月間・機能追加禁止ルール

> ローンチから3ヶ月間は機能追加・改良を原則禁止にする。
> 売れるかどうかはプロダクトではなく訴求で決まる。
> 完成度70%で出して、市場の反応を先に見る。

## 無料デモ設計（購入前の体験・最優先タスク）

**これがないと購入率は著しく下がる。「体験 → 欲しくなる」設計。**

### デモでできること（制限あり）

- 年収・支出・現在資産の3項目のみ入力可能
- 表示：FIRE到達年齢（推計値）のみ
- グラフはぼかし表示
- 「育休・産休の影響」「詳細シナリオ」は非表示

### デモ → 購入への導線

**メイン導線：デモ → 即購入（メール不要）**

```
デモ体験
  ↓ 「育休を反映したい」「最悪ケースを見たい」という欲求が生まれる
  ↓ 「育休を反映したい」「最悪ケースを見たい」という欲求が生まれる
  → デモ画面の直下に購入ボタン（note 有料記事 リンク）
  → 即購入 → アクセスコード取得
```

**補助導線：メール登録（メインに乗らなかった人向け）**

```
デモ体験 → 「今は購入しない」
  → メールアドレス入力（「使い方ガイドを無料送付」）
  → ステップメール（後述）
  → 購入を再検討
```

> FIREを本気で考えている層はメールニュースレターを嫌う傾向がある。
> デモで欲求が生まれた瞬間に購入できる設計が最優先。
> メール登録は「その場で決断できなかった人」の回収手段として位置づける。

### デモの技術実装

- Streamlit で公開（ブラウザのみで動作・インストール不要）
- デモ用とフル版は同一コードベース、パラメータで機能を分岐
- 工数目安：既存コードのStreamlit化 ＋ 制限ロジック追加で2〜3日

### 信頼感リスクへの対策（ケチらないポイント）

金融系はUIの"安心感"が購入を左右する。無名URL＋簡素デザイン＋個人運営の3点が重なると不安感が出る。

| 対策 | 内容 | 優先度 |
|------|------|--------|
| 独自ドメイン取得 | .com で十分（年1,000〜2,000円） | 高 |
| シンプルLP（1枚） | デモへの入口として「何ができるか」を1スクロールで説明 | 高 |
| データ非保存の明記 | 「入力データはサーバーに保存されません」をデモ上部に明記 | 高 |
| プロフィール写真・本名なし可 | 顔出し不要だが、X/noteとの連携を一貫させる | 中 |

> 独自ドメインとLP1枚は副収入モデルでもケチらない。
> 安心感はコンバージョン率に直接影響する。

| 商品単価 | 検討中 | 競合調査を踏まえて決定 |
| デモ → 購入転換率 | 1%（安全設計） | 無名個人プロダクト・金融系の保守的想定 |

> **月1,000デモは6ヶ月以内でも相当強い数字。**
> 「月3万円」は到達可能な上限ライン。標準目標は5件/月（月15,000円）で設計する。

| フェーズ | 期間 | 月間デモ数 | 購入数/月 | 月間収入 | 位置づけ |

| フェーズ | KPIの焦点 | 主なアクション |
|---------|----------|--------------|

### デモ数の現実的な積み上げ方

```

note 記事 1本（検索流入含む）→ 100件/月（3〜6ヶ月後）

Phase 1の150件 ≒ X + note1本 + 口コミ少々
```

> **転換率より流量が先に詰まる。Phase 1〜2は訴求改善、Phase 3は流量増加に集中。**

## 競合分析と差別化ポイント

| ツール | 特徴 | 本ツールの優位性 |
|--------|------|----------------|
| マネーフォワード ME | 家計管理は優秀。将来シミュレーションは簡易・無料 | 育休/産休/時短をそのまま入力でき、MCで確率まで出る |
| 金融庁シミュレーター | 非常に簡易。資産形成の入口用 | 支出・収入・年金を統合した包括モデル |
| Projection Lab（米国） | 高機能。月14ドル〜。英語のみ | 日本の税制・育休・NISA完全対応・買い切り |
| Excelテンプレート（無料ブログ） | 無料。MCシミュレーション非対応が多い | MC1,000回＋育休・産休の計算が自動化されている |

### 差別化の主軸：「確率思考で人生を設計する」

育休対応は機能差別化だが、**メッセージの軸は「確率思考」に置く**。

FIRE層は「不安を解消したい人」。抽象的な確率論よりも**感情に刺さる言い方**で届ける。

> **感情レベルに落とした刺さるコピー候補：**
> - 「『FIREできるか？』ではなく、『何%なら会社を辞める決断ができるか？』を計算する」
> - 「安心して会社を辞められる確率を、1,000通りの未来から計算する」
> - 「最悪ケースを先に見てから決断する。それだけで不安が消える」
> - 「リターンが悪い年が3年続いても大丈夫か？1,000回計算して確かめる」

育休対応だけを押し出すと「育休中の人しか刺さらない」という問題がある。
確率思考・最悪ケース可視化は**投資してFIREを目指す全員に刺さる訴求軸**。

育休は「このツールでできる具体例のひとつ」として位置づける。

### 訴求文の使い分け

| 場面 | 訴求軸 | 文言例 |
|------|--------|--------|
| X・note（認知） | 不安解消 × 確率思考 | 「安心して辞められる確率を計算した。結果、〇%だったので続けることにした」 |
| デモ → 購入（転換） | 育休・子供の具体例 | 「育休12ヶ月を正確に反映した上で確率計算できる唯一のツール」 |
| note 商品ページ | 感情 → 機能 → 安心 | 不安解消コピー → 確率思考 → 日本固有対応 → 購入案内の順 |

**無料ツールとの差別化訴求（購入ページ・記事で使う文言）：**

> 「無料ツールでできること：貯蓄額 ÷ 支出 × 25 でFIRE年数を計算する」
>
> 「このツールでできること：育休12ヶ月・時短2年・産前産後の収入変動を全部入れた上で、
> 1,000回シミュレーションして『FIRE確率〇%・最悪ケースでは資産が〇歳で底をつく』を出す」

## アクションプラン（6ヶ月ロードマップ）

### Phase 0：制作（1〜2週間）

**ゴール：最低限動くものを出す。完成度70%でよい。**

| 週 | タスク |
|----|--------|
| Week 1 | デモ版（Streamlit）作成・公開。デモ直下に購入ボタン設置 |
| Week 1 | note 有料記事下書き作成（説明文・スクショ3枚・FAQ） |
| Week 2 | フル版Webツール完成・アクセスコード認証実装 |
| Week 2 | メール補助導線設定（ステップメール3通） |
| Week 2 | note 記事1本を投稿。X 初回投稿。販売開始 |

### Phase 1：検証（1〜2ヶ月目）

**ゴール：「売れるか」の確認。目標：2〜3件/月。**

- X/noteで週1〜2回発信を継続（スクショ＋一言）
- デモ → 購入ボタンのクリック率・note転換率を計測
- **機能追加・改善は一切しない（3ヶ月禁止）**

```
判断基準：
  デモ利用があるのに購入ゼロ → 訴求文・価格を変える（プロダクトは変えない）
  デモ利用自体が少ない → 投稿頻度・タイトルを変える
  2件以上売れた → Phase 2 へ進む（改善フェーズ）
```

### Phase 2：改善（3〜4ヶ月目）

**ゴール：転換率の改善。目標：5〜6件/月。**

- 購入者にアンケート（1〜2問のみ）：「何が決め手でしたか？」
- 購入者の声を note 記事に追加
- デモの「制限解除ポイント」を見直し（欲求が生まれているか？）
- SEO記事の第1弾（1〜2本）を投稿

```
判断基準：
  5件/月を超えた → Phase 3 へ（流量増加フェーズ）
  3ヶ月合計5件未満 → 「誰が買ったか」をヒアリングしてペルソナを再定義
```

### Phase 3：拡大（5〜6ヶ月目）

**ゴール：流量を増やす。目標：10件/月（月3万円）。**

- SEO記事を追加（月1〜2本、合計5本を目指す）
- note有料記事（500円）でファンネル入口を増やす
- X フォロワー向けリツイート施策（スクショ投稿を強化）
- 購入者の事例をnoteで紹介（許可を得て）

### 判断フロー（3ヶ月後・6ヶ月後）

```
3ヶ月後：
  合計10件以上 → Phase 3 へ
  合計5件未満  → 訴求・ペルソナを再定義してから継続

6ヶ月後：
  月10件安定   → 現状維持 or サブスク化を検討
  月5〜9件     → SEO強化・X フォロワー拡大を継続
  月5件未満    → ペルソナを変更 or 一旦休止して別ツール検討
```

## 発信継続ルール（最大のボトルネックへの対策）

**戦略が完成しても、発信が止まれば終わる。このルールを守れるかが成否を分ける。**

### 3つの絶対ルール

| ルール | 内容 | 破れやすいタイミング |
|--------|------|---------------------|
| **週1投稿を6ヶ月守る** | 質より継続。反応がゼロでも投稿する | 反応が薄い2〜3週目 |
| **機能改善は月1回まで** | 「改善したい衝動」をルールで抑える | 反応があるとき・なさすぎるとき |
| **数字は月1回だけ見る** | 毎日見るとメンタルが削られ判断が狂う | ローンチ直後の不安期 |

### エンジニア気質の2つの罠

```
反応が薄い → 「理論・設計に問題がある」と思い込み、分析・改善に走る
反応がある → 「もっと機能を足せばもっと売れる」と思い込み、機能追加に走る

どちらも罠。正解は「改善よりも発信量を守ること」。
```

> このモデルは「大きく勝つ」ではなく「構造で積み上げる」設計。
> 毎週1投稿を6ヶ月続けられれば、3万円は時間の問題になる。
