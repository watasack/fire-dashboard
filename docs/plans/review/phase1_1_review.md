## 計算検証結果

### テストケース: grossIncome = 6,000,000 / employee / age=35

**手計算（本レビュー実施）:**

| 項目 | 手計算値 | 実装値 |
|------|----------|--------|
| 給与所得控除 | 1,640,000 | 1,640,000 |
| 給与所得 | 4,360,000 | 4,360,000 |
| 健保標準報酬月額 | 500,000 | 500,000 |
| 健康保険料 | 299,400 | 299,400 |
| 厚年標準報酬月額 | 500,000 | 500,000 |
| 厚生年金保険料 | 549,000 | 549,000 |
| 雇用保険 | 36,000 | 36,000 |
| 社保合計 | 884,400 | 884,400 |
| 課税所得 | 2,995,600 | 2,995,600 |
| 所得税（復興税込） | 206,303 | 206,303 |
| 住民税 | 304,560 | 304,560 |
| 合計税 | 1,395,263 | 1,395,263 |
| 手取り | 4,604,737 | 4,604,737 |

**課題指定の手計算との差異について:**

設問の手計算は社保合計 884,400 を 884,000 に丸めてから課税所得を計算しており、その連鎖で最終的に約81円の差（手取り: 4,604,656 vs 実装: 4,604,737）が生じている。これは実装の問題ではなく手計算側の簡略化（千円単位への丸め）による相違。

**実装は丸めを行わず浮動小数点のまま計算しており、数学的には正確。** 税務実務では1円単位で計算するため、実装の方針が正しい。

### 設計書との仕様差異

設計書 `phase1_gross_income.md` と実装との間に、`TaxBreakdown` 型の構造に差異がある:

| フィールド | 設計書仕様 | 実装 | 評価 |
|-----------|-----------|------|------|
| `healthInsurancePremium` | あり（健保料単体） | なし | 設計書はより細粒度だが、実装は `socialInsurance` に集約。後続フェーズで必要になれば拡張可 |
| `longTermCareInsurancePremium` | あり（介護保険料単体） | なし | 同上 |
| `pensionInsurancePremium` | あり（厚年料単体） | なし | 同上 |
| `totalSocialInsurance` → `socialInsurance` | `totalSocialInsurance` | `socialInsurance` | フィールド名が異なるが意味は同一 |

この差異は設計書が想定した「P2C・P3で標準報酬月額を再利用する」要件を考慮した拡張フィールドの省略であり、Phase 1.1 スコープとしては問題ない。`standardMonthlyRemuneration` は実装済みで後続フェーズに対応できる。

設計書は `residentTax = taxableIncome * 0.10`（均等割省略）と記述しているが、実装は `residentTax = taxableIncome * 0.10 + 5_000` と均等割5,000円を含めている。これは実態に即した正確な計算であり、設計書の「省略可」という記述に反するが、精度向上として問題なし（むしろ推奨）。

## 問題点

### 軽微な懸念（ブロッカーなし）

1. **`employmentType` が optional になっていない**（`Person` インターフェースで `employmentType?: EmploymentType`）のに、内部では `config.person1.employmentType ?? 'employee'` とフォールバックしている。型上は optional なので TypeScript が警告しないが、実態と整合している。

2. **`calculateNetIncome` 関数が設計書では定義されているが実装には存在しない**: 設計書 §「calculateNetIncome」で年次ループ用ラッパーを定義しているが、実装では直接 `calculateTaxBreakdown(...).netIncome` を呼んでいる。コードの見通しに影響はなく、関数が不要になったと判断したのであれば問題なし。

3. **配偶者の収入を合算してから単一の `calculateTaxBreakdown` に通している**: `totalIncome = person1収入 + person2収入` としてから1回の税計算を行っている。夫婦別々に税計算して手取りを合算する方が日本の税制に沿っているが、これは Phase 1.1 のスコープ外であり設計書の既知事項。後続フェーズ（P5・P6）での改善候補として記録しておく。
